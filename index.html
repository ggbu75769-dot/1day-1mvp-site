The problem

AI는 빨라졌지만, 무엇을 끝낼지는 여전히 흐려집니다.

하루 MVP가 막히는 지점은 코드 생산량이 아니라 범위, 검증 기준, 위험한 결정, 다음 판단이 흩어지는 순간입니다.

범위가 하루보다 큽니다

좋은 아이디어도 오늘 검증할 단위로 줄이지 못하면 실행이 늘어집니다.

완료 기준이 약합니다

패킷, 테스트, 출시 증거가 분리되면 무엇이 끝났는지 다시 확인해야 합니다.

속도와 안전이 섞입니다

결제, 개인정보, 배포, 외부 API, 도메인 결정은 자동으로 밀어붙일 일이 아닙니다.

How it works

아이디어에서 증거까지 여섯 단계로 닫습니다.

1Day-1MVP는 빌드 도구가 아니라 하루 실행을 끝까지 운영하는 작업 흐름입니다.

실험 선택

오늘 검증할 한 가지 아이디어와 대상 사용자를 고릅니다.

범위 잠금

오늘 할 일과 하지 않을 일을 분리하고 성공 기준을 적습니다.

작업 패킷 생성

Codex가 바로 실행할 수 있게 파일, 제약, 확인 명령을 묶습니다.

QA-Z 검증

코드, 문서, 테스트, 공개 파일 경계가 어긋나지 않는지 봅니다.

승인 대기

되돌리기 어려운 결정은 사람의 확인 전까지 멈춥니다.

증거와 리뷰

QA-Z quality gate

빠르게 만들수록 검증은 더 짧고 분명해야 합니다.

QA-Z는 변경이 다음 단계로 넘어가도 되는지 확인하는 품질 게이트로 설명합니다. 내부 판정 파일은 공개 사이트에 올리지 않습니다.

무엇을 확인하나요?

테스트 누락, 문서와 코드의 불일치, 공개하면 안 되는 값, 출시 파일 경계, 승인 항목을 함께 봅니다. 차단 항목이 있으면 고친 뒤 다시 확인합니다.

  • 문서와 실제 동작의 어긋남
  • 테스트나 출시 파일의 빠진 부분
  • 공개 사이트에 올라가면 안 되는 값
  • 승인 전 멈춰야 하는 결정

Safety by design

공개 사이트는 소개만, 실행 권한은 비공개로 둡니다.

GitHub Pages에는 정적 소개 파일만 배포합니다. 실제 실행, 승인, API 호출, 운영 기록은 공개 사이트에 두지 않습니다.

자동화가 혼자 결정하지 않는 일

결제, 개인정보, 실제 배포, 유료 외부 API, 도메인과 DNS, 저장소 공개, 라이선스 선택, 취약점 신고 채널은 승인 전까지 멈춥니다.

  • 정적 공개 사이트만 운영
  • Runtime API는 공개 배포하지 않음
  • 관리 콘솔은 비공개 환경에서 검증
  • 소스 공개와 라이선스는 승인 대기

FAQ

공개 페이지에서 바로 알아야 할 것

실제 서비스처럼 보이되, 현재의 공개 범위와 안전 경계를 과장하지 않습니다.

지금 바로 가입할 수 있나요?

아직 아닙니다. 파일럿이나 공개 데모가 준비되면 이 페이지에 다음 행동을 안내합니다.

이 페이지에서 데이터가 저장되나요?

아닙니다. GitHub Pages에 올라간 정적 소개 사이트이며 입력 폼이나 백엔드 연결을 두지 않았습니다.

누구에게 맞나요?

솔로 창업자, 인디 빌더, 작은 제품팀처럼 AI 에이전트와 함께 작은 실험을 끝까지 닫고 싶은 팀에 맞습니다.

하루 안에 항상 출시되나요?

모든 제품을 하루에 완성한다는 뜻은 아닙니다. 하루 안에 검증할 수 있는 작은 범위로 줄이고, 결과를 기록하는 것이 목표입니다.

왜 Runtime API를 공개하지 않나요?

공개 사이트는 소개 용도입니다. 실행 데이터, 승인 기록, 운영 API는 공개 정적 호스팅에 두지 않습니다.

소스 공개는 언제 열리나요?

저장소 공개, 라이선스 선택, 취약점 신고 채널 승인이 끝난 뒤에만 별도 절차로 진행합니다.
