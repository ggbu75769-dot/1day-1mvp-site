문제

왜 하루 MVP는 자꾸 밀릴까?

대부분의 MVP가 느려지는 이유는 개발력이 부족해서만은 아닙니다. 오늘 끝낼 범위, 검증 기준, 위험한 결정의 위치가 흐려질 때 실행이 밀립니다.

아이디어는 많은데 오늘 끝낼 범위가 없다

좋은 아이디어도 하루 안에 검증할 단위로 줄이지 못하면 계속 회의와 메모에 남습니다.

개발 지시는 길지만 검증 기준이 흐릿하다

무엇을 바꾸는지보다 무엇을 통과해야 하는지가 불분명하면 작업이 끝났는지 판단하기 어렵습니다.

출시했다고 말하지만 남은 증거가 없다

URL, 점검 결과, 막힌 이유, 다음 결정을 남기지 않으면 다음날 같은 질문부터 다시 시작합니다.

위험한 결정이 자동화 흐름에 섞인다

결제, 개인정보, 외부 API, 공개 릴리스 같은 결정은 속도보다 확인이 먼저입니다.

해결 방식

1Day-1MVP가 하는 일

하루 실행을 하나의 화면과 기록으로 묶어, 만들기 전과 만든 뒤의 판단이 끊기지 않게 합니다.

1Day-1MVP는 하루 단위의 MVP 실행을 하나의 흐름으로 정리합니다. 무엇을 만들지 정하고, Codex가 실행할 수 있는 작업 단위로 나누고, QA-Z로 점검하고, 사람이 확인해야 하는 결정은 따로 멈춰 세웁니다. 마지막에는 출시 URL, 검증 결과, 다음 결정을 남깁니다.

오늘 끝낼 범위

만들 것, 미룰 것, 성공 기준을 먼저 정합니다.

실행 가능한 작업 패킷

Codex가 바로 움직일 수 있도록 파일, 조건, 검증 명령을 묶습니다.

증거 기반 다음 결정

출시 결과와 막힌 이유를 남겨 다음 실행을 빠르게 고릅니다.

사람의 확인

자동화하지 않는 결정

1Day-1MVP는 모든 일을 자동으로 밀어붙이지 않습니다. 되돌리기 어려운 결정은 확인이 끝날 때까지 멈춥니다.

멈춰야 할 때 멈추는 구조

결제, 개인정보, 외부 API, 도메인/DNS, 공개 소스 릴리스처럼 되돌리기 어려운 결정은 사람이 확인하기 전까지 진행하지 않습니다.

  • 공개 소스 릴리스: 승인 대기
  • LICENSE: 승인 전까지 미생성
  • Runtime API: GitHub Pages에 배포하지 않음
  • 공개 사이트: 정적 소개 페이지로만 운영

FAQ

자주 묻는 질문

공개 사이트에서 볼 수 있는 것과 아직 공개하지 않은 것을 분명하게 나눕니다.

이 페이지에서 바로 앱을 사용할 수 있나요?

지금 이 페이지는 공개 소개 사이트입니다. 실제 프로젝트 관리 콘솔은 비공개 환경에서 실행됩니다. 공개 데모나 파일럿을 열 때는 이 페이지에 별도로 안내하겠습니다.

왜 백엔드를 공개하지 않았나요?

GitHub Pages는 정적 소개 페이지에만 사용합니다. 실행 데이터, 승인 기록, API 호출은 공개 사이트에 두지 않습니다.

소스코드는 언제 공개되나요?

저장소 공개, 라이선스 선택, 취약점 제보 채널 같은 항목은 승인이 끝난 뒤에만 진행합니다. 지금은 공개 소스 릴리스가 대기 상태입니다.

QA-Z는 왜 필요한가요?

빠르게 만들수록 놓치는 것이 생깁니다. QA-Z는 테스트, 문서, 보안, 릴리스 파일을 확인해서 믿고 다음 단계로 넘길 수 있는지 봅니다.

1Day-1MVP는 누구에게 맞나요?

아이디어가 많지만 하루 단위로 범위를 정하고, Codex 작업을 나누고, 결과를 보고 다음 결정을 내려야 하는 운영자에게 맞습니다.

하루 안에 정말 MVP가 나오나요?

모든 제품을 하루 만에 완성한다는 뜻은 아닙니다. 대신 하루 안에 검증 가능한 범위를 정하고, 만들고, 점검하고, 다음 결정을 내릴 수 있게 돕습니다.
